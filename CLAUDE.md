# Project Manager — agent notes

## Conventions

- **Next.js 16 App Router** with Turbopack (default). `middleware.ts` is renamed to `proxy.ts` (default export named `proxy`).
- **Async APIs**: `cookies()`, `headers()`, `params`, `searchParams` are all Promises. Always `await` them.
- **Supabase SSR** via `@supabase/ssr`. Server client in `lib/supabase/server.ts`; browser client in `lib/supabase/client.ts`; session refresh in `lib/supabase/proxy.ts` (invoked from `proxy.ts`).
- **Auth helpers**: `getSessionProfile`, `requireSession`, `requireStaff` in `lib/auth.ts`. Use these in server components & actions instead of querying `auth.users` directly.
- **Server actions** live in `app/actions/`. Validate input with `zod`. Use `revalidatePath` after mutations.
- **RLS** is the source of truth for access control. Don't hand-roll role checks in queries — RLS rejects rows the user can't see. The `requireStaff` helper is just a UX redirect.
- **UI primitives** in `components/ui/` (Button, Input, Dialog, Card, Badge, Avatar, EmptyState). Minimal — extend rather than reach for shadcn unless the need is repeated.
- **Design tokens** in `app/globals.css` under `@theme`. Use `bg-brand-500`, `text-foreground`, `border-border`, etc. instead of raw Tailwind colors so the look stays consistent.

## Database

- Migrations: `supabase/migrations/NNNN_name.sql`.
- Apply via Supabase MCP `apply_migration` or `supabase db push`.
- Regenerate types: `supabase gen types typescript --project-id <ref> > lib/db/types.ts`.
- After any DDL, run `get_advisors` (security + performance) and fix WARNs.

## Projects & templates — model

- **New project dates**: `projects.start_date` = the CRM's "Projected Start Date" (pulled through the dashboard; `normalizeDashboardProject` accepts `start_date`/`projected_start_date`/`projected_start`). On the template-copy path (`duplicateProject`), the schedule is anchored on the **Job Start milestone** so it lands exactly on `start_date` (falls back to the earliest dated item if the template has no dated Job Start); every item shifts by the same delta. Blank projects stamp their Job Start milestone with `start_date` too. There is **no** `target_completion_date` — the column was dropped (0081); a job's projected finish is the Substantial Completion milestone + health banner, not a project field.
- **Smart templates** (0042): template `schedule_items`/`decisions` carry `template_tags text[]`; `projects.attributes jsonb` stores the boolean answers captured at creation. An item copies only when `matchesTemplateTags` — every tag matches (`walkout` needs true, `!walkout` needs false; empty = always copy). Helpers in `lib/template-tags.ts`. Template tags are surfaced as purple chips on the schedule list view **only when `projects.is_template`** — inert and hidden once copied into a real job.
- **Either/or tag groups**: staff group tags into mutually-exclusive choices in Settings → Template tags (avatar menu, staff-only, `/settings/template-tags`), stored as JSON in `app_settings` key `template_tag_groups` (`getTemplateTagConfig`/`saveTemplateTagConfig` in `app/actions/settings.ts`). A **required** group renders as a single-select in the create/duplicate questionnaire (`TemplateOptionsFields`) and blocks "Create project"/"Duplicate" until answered (`TemplateOptionsValue.valid`); picking an option sets that tag true and its siblings false via `attributesWithGroupSelections`, riding on the same boolean-attributes matcher. Enforcement is client-side (staff-only flows).

## Decisions module — model

- One page for both `change_order` and `selection` kinds — they share table `decisions` with a `kind` enum and a per-project sequential `number`.
- Workflow: `draft` → `pending_client` → `approved` | `rejected`. Clients can only see decisions once they leave `draft`.
- `decision_followup_templates` are per-decision to-do templates the user defines while drafting. On the `approved` transition, `materializeFollowups` in `app/actions/decisions.ts` creates real `schedule_items (kind='todo')` with `source_decision_id` set, plus assignments and in-app notifications for staff assignees. Re-approval is idempotent via the `decision_followup_materializations` junction (PK decision_id+template_id, claim-then-insert) — NOT title matching.
- Due dates can be **linked to a schedule item** (0072): `due_anchor_schedule_item_id` + `due_anchor` (start|end) + `due_anchor_offset_days` on `decisions`, all-or-nothing triple. `due_date` stays canonical — `saveDecision` computes it from the anchor on save, and the `schedule_items` trigger `sync_decision_due_anchors` refreshes it on any date move (covers cascades, bulk shifts, AI apply) and unlinks on delete, freezing the last date. The same trigger clears follow-up-template anchors on delete (the 0035 SET NULL + triple-check combo used to make those deletes fail). Cross-project copies drop the link; `duplicateProject` remaps it through the cloned schedule.
- **PGRST201 warning**: decisions↔decision_choices (via `decision_id` + `selected_choice_id`) and decisions↔schedule_items (via `due_anchor_schedule_item_id` + `source_decision_id`) each have TWO relationships — always FK-hint embeds between these pairs (e.g. `decision_choices!decision_choices_decision_id_fkey(...)`), or PostgREST 300s and the embed comes back empty.
- `decision_comments`: staff full access; clients in `project_members` for the project can both READ and INSERT (RLS enforces `author_id = auth.uid()`). Trades have no access.
- Decision attachments share the `project-files` Storage bucket with daily logs. The storage RLS policy was extended in 0004 to allow client read for both daily-log AND decision attachments via signed URLs.
- **Past-due gate** (0074): the `client_decide_decision` RPC rejects APPROVE when `due_date < current_date` (decline stays allowed). The drawer's client footer disables Approve and offers "Request due-date reset" → `requestDueDateReset` (client-callable; leaves a decision_comment + notifies staff in-app/email via the admin client). Staff reset = just edit the due date.
- **Decision assignments** (0075): `decision_assignments` mirrors schedule_assignments (exactly one of profile/company/role, unique nulls-not-distinct). Staff assign selections in the drawer; `saveDecision` wipe-and-reinserts from `input.assignments`. Trades gain READ on assigned non-draft decisions + their choices/attachments (helper `trade_sees_decision`, storage policy extended); surfaced on `/my-assignments` under "Selections". Cost items stay staff-only. 0078 adds `projects_trade_read` (helper `trade_sees_project`: schedule assignment, role membership, or decision assignment) — without it trades can't read the projects row, so `projects!inner` embeds silently drop rows and `/projects/{id}/...` 404s.
- **Disclaimer** (0077): org-wide `app_settings` key `decision_disclaimer` (read-all-authenticated, staff write via `saveDecisionDisclaimer`). Edited in a box at the top of the decisions page; rendered at the bottom of every decision the CLIENT views (and in preview).
- **Client preview**: the drawer's "Preview as client" toggle (staff, edit mode) re-derives isClient/canEdit locally; approve/decline/request buttons render disabled. No impersonation — same staff session, UI-only.
- **Catalog link** (0076): `decision_cost_items.catalog_item_id` (bare uuid — cross-DB, no FK) + `catalog_item_code` snapshot point at `catalog_items` in the HH-SpecMagician Supabase project. `lib/supabase/specmagician.ts` (clone of the CRM client pattern; `SPECMAGICIAN_SUPABASE_URL`/`_SERVICE_ROLE_KEY`, null when unset) + `searchCatalogItems` in `app/actions/catalog.ts` power the per-line picker in both cost-item editors.
- **Export**: staff "Export" button on the decisions page builds a 3-sheet .xlsx (Decisions / Choices / Line items) client-side via the dependency-free writer `lib/export/xlsx.ts` (stored-zip + inline strings — no npm dep).
- **Bulk copy**: staff multi-select decisions (checkbox column) → floating bar → `bulkCopyDecisions` (loops `copyDecision` per id: fresh draft, new number, approvals/choices reset, cross-project anchors dropped, assignments copied verbatim).

## Daily Logs module — model

- `daily_logs(visibility = 'internal'|'client')` — visibility is rendered prominently in the UI (left border + badge). `internal` is hidden from the future client portal; `client` is shown.
- Subs/vendors that were on site live in `daily_log_subs_on_site (daily_log_id, company_id, notes)`.
- Files in `daily_log_attachments` reference Supabase Storage objects in bucket `project-files`, with key `projects/{project_id}/daily-logs/{random}.{ext}`. Bucket is private. Server-side actions issue 1-hour signed URLs via `getSignedUrls` in `app/actions/daily-logs.ts`.
- Browser uploads go directly to Storage with the user's JWT (RLS policy `project_files_staff_all`). The action `saveDailyLog` then records the path in `daily_log_attachments`.
- Clients can `select` from `daily_logs` only when `visibility = 'client'` AND they're in `project_members` for that project — enforced by RLS. Trades have no access to daily logs.
- **AI client-update composer**: "Draft client update" (staff, `daily-logs-client.tsx`) calls `draftClientUpdate` in `app/actions/daily-logs.ts`, which gathers the last week's INTERNAL logs plus recently-completed and upcoming work items and runs one Claude call (`lib/ai/client-update.ts`, single-shot like the insurance extractor) to rewrite them into a homeowner-friendly note. The draft prefills the daily-log drawer preset to `visibility = 'client'`; nothing publishes until the staffer saves via `saveDailyLog` (which fires the `daily_log.published` dashboard webhook).

## Schedule/To-Dos module — model

- `schedule_items` is a single table; `kind` is `'work'` or `'todo'`. A to-do nests under a work item via `parent_id`. Standalone to-dos have `parent_id = null`.
- Every project carries two protected milestone work items (`schedule_items.milestone`: `'job_start'` | `'substantial_completion'`, unique per project). They move/complete/link like any work item but can never be deleted (DB trigger `protect_schedule_milestones` + app-layer guards). `ensureProjectMilestones` runs on every project-creation path and ADOPTS existing work items titled "Job Start"/"Substantial Completion" (case-insensitive — the Template ships with them) before creating new rows (0070); the health banner offers a fallback button.
- **Baseline** (0069): "Set baseline" on the schedule page (`setScheduleBaseline` → RPC `set_schedule_baseline`) copies each work item's current dates into `baseline_start_date`/`baseline_end_date` and stamps `projects.baseline_set_at`. Pre-baseline, work items can't be marked `complete` (to-dos can). Post-baseline, any date change to a work item requires a reason (`MoveReasonDialog` popup → `schedule_delays` row); the AI apply path auto-logs under `other`. Work items created post-baseline get their initial dates stamped as baseline; `duplicateProject` copies milestones but nulls baselines.
- **Health banner** (`components/schedule/health-banner.tsx`, math in `lib/schedule/health.ts`): job duration = Job Start → Substantial Completion, vs baseline, with a 30-day buffer. Green "X Days Remaining in Buffer" (slip ≤ 30), yellow "X Days Late" (1–15 late), red (16+ late). While Substantial Completion is incomplete its projected date is clamped to today, so a stale schedule slips one day per day.
- Predecessors live in `schedule_predecessors`. FS is default. Cycle detection lives in `lib/schedule/scheduling.ts:wouldCreateCycle`.
- Cascading dates after a move: `cascadeFromPredecessors` returns successor updates; the server action applies them in a loop.
- **Recurring to-dos = roll-on-complete**: `recurrence_rule` jsonb ({freq daily|weekly|biweekly|monthly, interval, until, count}) lives on exactly ONE row of a series. Completing that row (any path: checkbox, sheet, bulk, onsite quick-update, AI apply — all call `lib/schedule/roll-recurrence.ts:rollRecurringTodo`) strips the rule off the completed row and inserts the next occurrence (due advanced past today along the anchor cadence via `lib/schedule/recurrence.ts:rollRecurrence`, checklist reset, assignments copied, count decremented). No rows are pre-created and nothing is virtual; `expandRecurrence` exists for future calendar use only. A rule without a due date is inert (the dialog blocks saving that). The onsite path passes `anchorDueOverride` because it rewrites due_date while completing.
- Priority flags: `PriorityBadge` renders ONLY for `high` (low/medium show nothing anywhere — filter/sort still use all three).
- Assignments can target a profile (internal staff), a company (sub/vendor), or a role — exactly one of the three must be non-null (0054). The bulk bar assigns/unassigns people AND roles (`bulkAssignRoleToScheduleItems` etc.).
- **Bulk copy to job**: the list view's bulk bar copies selected work items + to-dos to another project (`bulkCopyScheduleItems`): parent/child + predecessor edges survive when both ends are selected, statuses reset, checklists reset, assignments verbatim, milestones skipped, baseline stamped if the target is baselined.
- **Project History tab** (0073): `project_history` (staff-read, trigger-only writes via SECURITY DEFINER `record_project_history`) audits create/update/delete on schedule_items, decisions, daily_logs, project_files, project_payments, bid_packages, purchase_orders, project_members, project_role_members, schedule_assignments (+ projects updates) with per-field diffs and a `txid` batch key. `project_id` is a bare uuid (no FK) so project-delete cascades can't deadlock on it. Rendered at `/projects/[id]/history` (tab right of Roles, hidden from client/trade; latest 500, batched by txid).

## Bid Requests module — model

- `bid_packages` (per-project sequential `number`, statuses `draft → sent → awarded | closed`) with `bid_package_line_items` (cost-coded pricing structure, no unit cost) or `flat_fee` mode. Subs' pricing lives in `bid_line_item_quotes` / `bid_recipients.flat_total`.
- One `bid_recipients` row per invited company, with statuses `invited → submitted | declined → awarded` and an unguessable `token` — the sub's only credential. Public page `/bid/{token}` (no login); mutations in `app/actions/bid-public.ts` run on the **admin client** with compare-and-swap status guards. **No anon RLS policies** — the anon key can never touch bid/PO tables. Revocation = nulling the token (close/unrelease).
- Trade-role subs also see their own company's rows at `/my-bids` (RLS `br_trade_read` / `bp_trade_read`; never competitors' quotes). Cards link to the same token pages — one response UI.
- Staff actions in `app/actions/bids.ts`. "Revise & re-request" (not silent edit) wipes quotes and resets non-declined recipients on a released package. `award_bid` RPC atomically awards + optionally creates a draft PO pre-filled from the winning quotes (`source_bid_recipient_id` links back).
- Send/notify via `sendEmail` + `sendQuoSms`, respecting `companies.notifications_enabled`. Sub submissions notify staff by email + in-app notification (inserted via admin client).

## Purchase Orders module — model

- `purchase_orders` (per-project `number` + optional `custom_number`, statuses `draft → released → approved | declined`, plus `void`; `work_complete` is an independent flag) with `po_line_items` (cost-coded, with unit_cost) or flat fee. v1 has **no payments/bills** — that stays in QuickBooks/Adaptive.
- Release mints a `token` and emails/SMSes the sub a public `/po/{token}` link; the sub approves with a typed signature + disclaimer checkbox (`approved_by_profile_id` null) or declines with a reason. Staff can approve on behalf (`staffApprovePurchaseOrder`, profile id recorded). Unrelease pulls it back to draft, revokes the token, and clears approval state; void keeps the record but kills the link.
- Structural edits are draft-only — released POs must be unreleased first.
- Trade portal page `/my-pos` (RLS `po_trade_read`, own company + non-draft only).
- Approved POs roll up as **Committed costs** by cost code on the project Pricing tab — staff with `profiles.financial_access` only, never clients.
- Numbering RPCs `next_bid_package_number` / `next_po_number` use per-project advisory locks (hash args 1 / 2; decisions use 0). `award_bid` allocates PO numbers under the same lock key as `next_po_number`.

## Subcontractor insurance module — model

- `insurance_documents` (one per ingested COI file) + `insurance_policies` (one per policy parsed off it; enum `insurance_type`: general_liability | workers_comp | auto | umbrella). "Current" policy for a company+type = latest `expiration_date`; older rows are history. Staff-only RLS on both.
- Three ingestion paths, all funneling into `lib/insurance/ingest.ts` (admin client): the Resend inbound webhook `/api/inbound/insurance` (verify with `RESEND_WEBHOOK_SECRET`, download attachments via `resend.emails.receiving.attachments`), the public tokenized sub upload (`/insurance-upload/{companies.insurance_upload_token}` → POST `/api/insurance-upload`), and staff manual upload (browser → Storage, then `processStoredInsuranceDocument`).
- Extraction: `lib/insurance/extract.ts` reads the PDF/image with Claude (`claude-opus-4-8`, structured outputs). Result is stored on `insurance_documents.extraction` so the review queue can assign an unmatched doc to a company without re-running the model. Company auto-match: sender email exact match, then unambiguous name match; anything fuzzy → `status='needs_review'`.
- Reminders: daily cron `/api/cron/insurance-reminders` (auth `CRON_SECRET`) emails each company once per policy when a CURRENT policy expires within 7 days, with their upload link, then stamps `reminder_sent_at`. Respects `companies.notifications_enabled`; the staff "Send request" button does not (explicit click wins). Gated by a global kill switch: the cron no-ops unless `INSURANCE_REMINDERS_ENABLED === "true"` (OFF by default so automatic emails don't go out before the site is live; the manual button is never gated).
- Only companies with `companies.status = 'Approved for Use'` (case-insensitive; helper `lib/insurance/requirements.ts:companyRequiresInsurance`) are REQUIRED to carry insurance — the cron skips everyone else and the dashboard neither lists them by default nor flags their missing coverage ("Show all statuses" reveals them).
- Files live in the `project-files` bucket under `companies/insurance/…` (staff storage RLS already covers it; server paths use the admin client).
- Staff UI: `/companies/insurance` — coverage table (GL/WC required; auto/umbrella tracked), review queue, manual upload, send-request.

## AI smart-update agent — model

- Server action `runAgentTurnAction` in `app/actions/ai-agent.ts` wraps a manual Claude tool-use loop in `lib/ai/agent.ts`. Model is `claude-sonnet-5` with adaptive thinking on (a tool loop needs the model to keep reaching for tools) and `max_tokens` 8192 so thinking + tool calls + summary don't truncate. The tools + system prefix is prompt-cached (a `cache_control` breakpoint on the system block, stable within a turn). Requires `ANTHROPIC_API_KEY` env var (set in Vercel + `.env.local` for dev) — action returns a typed `error` result if the key is missing, never throws.
- Plan-then-approve flow: the agent's `propose_*` tools record mutations into a per-turn array but DON'T execute anything. Only `applyPlanAction` actually writes to the DB, and it runs under the caller's session so RLS still gates writes. Each plan carries a server-generated `plan_id`; `applyPlanAction` claims it in `ai_plan_applications` (unique) BEFORE applying, so a re-apply (double-click, or the walkthrough's blind retry) returns the first apply's stored results instead of duplicating writes / re-texting subs. That table is also the audit trail (proposed `mutations` + `results` per apply). If a turn is cut short (`stop_reason` max_tokens, or the iteration cap), the plan result carries `incomplete` and `PlanCard` shows an amber warning.
- Adding a new mutation kind takes five changes: extend `ProposedMutation` in `lib/ai/types.ts`, add the `propose_*` tool definition + handler in `lib/ai/agent.ts`, add the apply path in `lib/ai/apply.ts`, add a case in the shared plan-row renderer `components/ai/plan-review.tsx:MutationRow`, and extend the Zod `MutationSchema` in `app/actions/ai-agent.ts`. Mutation kinds: `add_checklist_item`, `create_todo` (optional assignee), `create_work_item`, `create_decision`, `add_decision_followup`, `append_daily_log` (optional photos + `subs_on_site`), `update_schedule_item_status`, `update_schedule_item`, `update_decision_status`, `assign_schedule_item`, `send_sms`, `send_bid_reminder`. Read-only tools also answer questions (reporting mode): `list_staff`, `get_schedule_health`, `list_schedule_delays`, `list_daily_logs`, `list_bid_packages`/`get_bid_package`, `list_purchase_orders` (status only — never dollar amounts; PO costs are `financial_access`-gated at the app layer, not RLS).
- **Help-desk mode**: the agent also explains how the APP itself works ("how do I set a baseline?", "what's a change order vs a selection?"). Knowledge lives in `lib/ai/help-content.ts` as staff-facing `HELP_TOPICS` (one per module — product docs, no RLS/migration jargon). `helpCatalog()` renders the id+title+summary of every topic into the (cached) system prompt so the model always sees the full menu; the read-only `search_help_topics` tool (`{ query?, topic_ids? }`, ranked substring match in `searchHelpTopics`) returns the full `body` of matched topics on demand. Adding/updating a topic is a one-file edit — keep bodies in sync with the real UI. Help answers propose nothing and never append a daily log (same guard as reporting mode). The system prompt's "three hats" intro routes a message to help-desk / reporting / field-notes by intent.
- Field-notes mode: dictated/typed site notes map to schedule updates, to-dos (optionally assigned to staff via `list_staff` or a sub/vendor), assignments on existing items (`assign_schedule_item`, in-app notification to a staff assignee), an SMS to a sub (`send_sms`, sent via `lib/quo.ts` with the recipient re-resolved from `companies.phone` at apply time), a bid reminder to invited-but-unresponded recipients (`send_bid_reminder`, reuses each recipient's existing token), and ALWAYS one `append_daily_log` per affected project (appends to, or creates, the internal daily log for that date; records any on-site subs in `daily_log_subs_on_site`). The browser sends its local date as `today` so "today" means the user's day, not UTC's. The chat dialog has a Web Speech API mic button for phone dictation (shims shared via `lib/speech/web-speech.ts`), and auto-scopes to the project when opened from a `/projects/[id]` route (`context.mode: "page"` — a dismissible "Scoped to …" chip; no photos, unlike the onsite `mode`).
- Onsite walkthrough (`components/onsite/walkthrough.tsx`, top of the project Onsite tab): voice-memo segments + photos feed the same agent with a trusted `context: { project_id, attachments }` on `runAgentTurnAction`. The server re-resolves the project under the caller's RLS session and adds an "On-site context" section to the system prompt (explicit carve-out to the IDs-from-tool-results rule for that one server-verified id). Photos never go to the model — after the plan returns, the action injects them into its `append_daily_log` for that project (synthesizing one if missing) so they deterministically land as `daily_log_attachments`; apply rejects storage paths outside `projects/{project_id}/daily-logs/`. Review is per-item checkboxes + a two-tap confirm for destructive rows (the global dialog keeps its all-or-nothing typed-"apply" gate). Drafts (segments + uploaded photo paths) persist in localStorage per project; photos orphaned by a discarded draft or an unchecked log row are accepted as unreferenced private-bucket objects (v1 policy).

## Communications / Quo phone — model

- Every outbound text/call and inbound event is logged to `communications` (see the Daily Logs/Bids modules for feed rendering). `sendQuoSms` in `lib/quo.ts` is the single SMS sender; `app/api/inbound/quo/route.ts` ingests the OpenPhone webhook (`message.received/.delivered`, `call.completed`, `call.recording.completed`).
- **Per-user Quo numbers** (0081): staff can be assigned their own Quo (OpenPhone) number on the Team page — stored as `profiles.quo_phone_number_id` ("PN…", the API `from`) + `profiles.quo_phone_number` (E.164), 1:1 via partial-unique indexes. `sendQuoSms` resolves the sending number from `senderProfileId ?? log.sent_by` (which every call site already sets to the acting staffer) → their stored number → the shared `QUO_FROM_NUMBER` fallback. So automated texts (bid/PO/assignment/manual/hub-reply/AI-apply) go out from the initiator's own number with **zero call-site changes** — they all pass `log.sent_by`. An explicit `opts.from` still overrides.
- The Team picker is populated by `listQuoPhoneNumbers()` (`GET /v1/phone-numbers`); empty when `QUO_API_KEY` is unset, and the dialog then only offers the shared number. `updateProfile` maps a 23505 to a friendly "already assigned" error; a non-staff role clears the assignment.
- **Native attribution**: the webhook matches each event's business-side number (`phoneNumberId` first, then E.164) to a profile via `staffProfileForQuoNumber`. Outbound rows get `sent_by` = that staffer (so texts they type and calls they dial in the Quo app show their name in the feed, which renders `sent_by` for outbound); inbound rows keep the counterparty as author but stash the line owner in `meta.quoStaffProfileId`. App-sent texts already hold `(source,provider_id)` with richer attribution, so the webhook's `ON CONFLICT DO NOTHING` leaves them untouched. Note: the Quo/OpenPhone API can only *send texts* and *track* calls (via webhook) — it cannot *place* calls.
- **Job attribution & the global hub**: `matchCounterparty` (`lib/comms/match.ts`, via the `match_contacts_by_phone/email` service-role RPCs in 0067) resolves an inbound number/address to a project when it can (project client contact → profile → company's single active engagement). Quo phone traffic is **never queued for manual placement**: the webhook always writes `status='logged'`, so it auto-files to a job only when the matcher is confident (a single-job sub) and otherwise stays global-only. A conversation can span multiple jobs, so this is intentional — staff don't have to file anything. The global staff hub `/communications` (`GlobalCommunicationsClient`) is the searchable home for all channels (shows everything except `status='ignored'`); unfiled rows offer a quiet optional "File to job" (`assignCommunication`) / "Dismiss" (`ignoreCommunication`) — there is no nagging needs-review queue. Job-level `/projects/[id]/communications` still shows only `project_id=this AND status='logged'`. (Inbound email uses the same matcher but prefers the `+p_<project_id>` reply plus-tag; unmatched email keeps `status='needs_review'`, which now renders identically to any other unfiled row in the hub.)

## Workflow

- `npm run dev` to develop; `npx tsc --noEmit` + `npx eslint .` for checks; `npx next build` to verify production build.
- Verify UI changes by running the app and clicking through — type checks don't catch UI regressions.
- Commit early. Cut feature branches from `main` and open PRs against `main` — that's the active integration branch where all recent work lands. (The old `claude/buildertrend-replacement-JsF4R` branch is stale and far behind `main`; don't target it.)
- **Pull requests**: open as ready-for-review (not draft) so CodeRabbit reviews immediately, and subscribe to PR activity right after opening.

## User environment

- Brandon does **not** have a local development environment — no `git clone`, no Node.js, no `npm`, no local editor running the project. All code edits happen either via the GitHub web UI or through this remote Claude Code session (which commits + pushes from its ephemeral container). All testing happens on Vercel preview / production deploys, not localhost.
- Skip these patterns when giving instructions:
  - "open VS Code" / "edit `.env.local`" / "run `npm run dev`" / "git clone" / "git checkout"
- Use these patterns instead:
  - Code changes: tell Brandon what you're about to change, edit the file from this session, commit + push, and Vercel auto-deploys a preview.
  - Env vars: set in the **Vercel dashboard** (Settings → Environment Variables), then redeploy. Don't reference `.env.local`.
  - Testing: have Brandon visit the Vercel preview URL (for branch deploys) or production URL.
- Brandon runs ad-hoc shell commands (like DNS lookups) in **PowerShell** on Windows. Use PowerShell syntax for those (`Resolve-DnsName` not `dig`, `$env:FOO="bar"` not `export FOO=bar`, `Copy-Item` not `cp`).
