# Project Manager ‚Äî agent notes

## Conventions

- **Next.js 16 App Router** with Turbopack (default). `middleware.ts` is renamed to `proxy.ts` (default export named `proxy`).
- **Async APIs**: `cookies()`, `headers()`, `params`, `searchParams` are all Promises. Always `await` them.
- **Supabase SSR** via `@supabase/ssr`. Server client in `lib/supabase/server.ts`; browser client in `lib/supabase/client.ts`; session refresh in `lib/supabase/proxy.ts` (invoked from `proxy.ts`).
- **Auth helpers**: `getSessionProfile`, `requireSession`, `requireStaff` in `lib/auth.ts`. Use these in server components & actions instead of querying `auth.users` directly.
- **Server actions** live in `app/actions/`. Validate input with `zod`. Use `revalidatePath` after mutations.
- **RLS** is the source of truth for access control. Don't hand-roll role checks in queries ‚Äî RLS rejects rows the user can't see. The `requireStaff` helper is just a UX redirect.
- **UI primitives** in `components/ui/` (Button, Input, Dialog, Card, Badge, Avatar, EmptyState). Minimal ‚Äî extend rather than reach for shadcn unless the need is repeated.
- **Design tokens** in `app/globals.css` under `@theme`. Use `bg-brand-500`, `text-foreground`, `border-border`, etc. instead of raw Tailwind colors so the look stays consistent.

## Database

- Migrations: `supabase/migrations/NNNN_name.sql`.
- Apply via Supabase MCP `apply_migration` or `supabase db push`.
- Regenerate types: `supabase gen types typescript --project-id <ref> > lib/db/types.ts`.
- After any DDL, run `get_advisors` (security + performance) and fix WARNs.

## Projects & templates ‚Äî model

- **New project dates**: `projects.start_date` = the CRM's "Projected Start Date" (pulled through the dashboard; `normalizeDashboardProject` accepts `start_date`/`projected_start_date`/`projected_start`). On the template-copy path (`duplicateProject`), the schedule is anchored on the **Job Start milestone** so it lands exactly on `start_date` (falls back to the earliest dated item if the template has no dated Job Start); every item shifts by the same delta. Blank projects stamp their Job Start milestone with `start_date` too. There is **no** `target_completion_date` ‚Äî the column was dropped (0081); a job's projected finish is the Substantial Completion milestone + health banner, not a project field.
- **Role assignments carry over from the template**: `duplicateProject` copies the source's `project_role_members` (the "who fills each role" map ‚Äî Project Manager, Site Superintendent, every trade role) verbatim into the new job. Roles are org-wide, so `role_id` copies directly; only `project_id` changes and the profile/company assignee is copied as-is. (Role-*based schedule assignments* already carried forward separately.) So every job built from the Template inherits the same people/companies in the same roles.
- **Purchasing carries over from the template**: `duplicateProject` also copies purchase orders and bid packages as **fresh drafts** ‚Äî numbers re-allocated 1..N in source order, all workflow state reset (tokens/release/approval/awards cleared, `work_complete` off), line items + vendor kept. `void` POs are skipped; a PO's `source_decision_id` remaps through the cloned decisions while `source_bid_recipient_id` drops (the copied bid round never happened on the new job). Bid recipients copy as the invite list (`invited`, no token ‚Äî tokens mint at Send); quotes never copy. Attachments blob-copy like decision attachments (failures skip, counted in `purchasingAttachmentsFailed`, surfaced in the duplicate toast); 0095 Files-tab links (`project_file_id`) drop because `project_files` aren't cloned. Dates (`approval_deadline`, bid `due_date`) shift with the schedule. Any child-insert failure mid-clone deletes the new project shell (children cascade) before rethrowing, so the project number can be retried.
- **Status mirrors the CRM at creation**: both the blank/dashboard create path (`createProject`) and every template-copy path (`duplicateProject`) resolve `status` from the CRM by `project_number` via `getCrmProjectStatus` (`lib/supabase/crm.ts`, same `crmStatusToEnum` mapping as `syncProjectsFromCrm`) and store the verbatim `crm_status` word. So a new job matches the dashboard from birth instead of defaulting to "In Work" until the next "Sync from CRM". The CRM wins over the form's status / `override_status`; falls back to the form value when the CRM isn't configured or has no matching row.
- **Smart templates** (0042): template `schedule_items`/`decisions` carry `template_tags text[]`; `projects.attributes jsonb` stores the boolean answers captured at creation. An item copies only when `matchesTemplateTags` ‚Äî every tag matches (`walkout` needs true, `!walkout` needs false; empty = always copy). Helpers in `lib/template-tags.ts`. Template tags are surfaced as purple chips (`components/template-tag-badges.tsx`) on the schedule list view AND the decisions table **only when `projects.is_template`** ‚Äî inert and hidden once copied into a real job. Both editors (schedule dialog + decision drawer) expose the tags field to staff everywhere.
- **Either/or tag groups**: staff group tags into mutually-exclusive choices in Settings ‚Üí Template tags (avatar menu, staff-only, `/settings/template-tags`), stored as JSON in `app_settings` key `template_tag_groups` (`getTemplateTagConfig`/`saveTemplateTagConfig` in `app/actions/settings.ts`). A **required** group renders as a single-select in the create/duplicate questionnaire (`TemplateOptionsFields`) and blocks "Create project"/"Duplicate" until answered (`TemplateOptionsValue.valid`); picking an option sets that tag true and its siblings false via `attributesWithGroupSelections`, riding on the same boolean-attributes matcher. Enforcement is client-side (staff-only flows).

## Decisions module ‚Äî model

- One page for both `change_order` and `selection` kinds ‚Äî they share table `decisions` with a `kind` enum and a per-project sequential `number`.
- Workflow: `draft` ‚Üí `pending_client` ‚Üí `approved` | `rejected`. Clients can only see decisions once they leave `draft`.
- `decision_followup_templates` are per-decision to-do templates the user defines while drafting. On the `approved` transition, `materializeFollowups` in `app/actions/decisions.ts` creates real `schedule_items (kind='todo')` with `source_decision_id` set, plus assignments and in-app notifications for staff assignees. Re-approval is idempotent via the `decision_followup_materializations` junction (PK decision_id+template_id, claim-then-insert) ‚Äî NOT title matching.
- Due dates can be **linked to a schedule item** (0072): `due_anchor_schedule_item_id` + `due_anchor` (start|end) + `due_anchor_offset_days` on `decisions`, all-or-nothing triple. `due_date` stays canonical ‚Äî `saveDecision` computes it from the anchor on save, and the `schedule_items` trigger `sync_decision_due_anchors` refreshes it on any date move (covers cascades, bulk shifts, AI apply) and unlinks on delete, freezing the last date. The same trigger clears follow-up-template anchors on delete (the 0035 SET NULL + triple-check combo used to make those deletes fail). Cross-project copies drop the link; `duplicateProject` remaps it through the cloned schedule.
- **PGRST201 warning**: decisions‚Üîdecision_choices (via `decision_id` + `selected_choice_id`) and decisions‚Üîschedule_items (via `due_anchor_schedule_item_id` + `source_decision_id`) each have TWO relationships ‚Äî always FK-hint embeds between these pairs (e.g. `decision_choices!decision_choices_decision_id_fkey(...)`), or PostgREST 300s and the embed comes back empty.
- `decision_comments`: staff full access; clients in `project_members` for the project can both READ and INSERT (RLS enforces `author_id = auth.uid()`). Trades have no access.
- Decision attachments share the `project-files` Storage bucket with daily logs. The storage RLS policy was extended in 0004 to allow client read for both daily-log AND decision attachments via signed URLs.
- **Past-due gate** (0074): the `client_decide_decision` RPC rejects APPROVE when `due_date < current_date` (decline stays allowed). The drawer's client footer disables Approve and offers "Request due-date reset" ‚Üí `requestDueDateReset` (client-callable; leaves a decision_comment + notifies staff in-app/email via the admin client). Staff reset = just edit the due date.
- **Decision assignments** (0075): `decision_assignments` mirrors schedule_assignments (exactly one of profile/company/role, unique nulls-not-distinct). Staff assign selections in the drawer; `saveDecision` wipe-and-reinserts from `input.assignments`. Trades gain READ on assigned non-draft decisions + their choices/attachments (helper `trade_sees_decision`, storage policy extended); surfaced on `/my-assignments` under "Selections". Cost items stay staff-only. 0078 adds `projects_trade_read` (helper `trade_sees_project`: schedule assignment, role membership, or decision assignment) ‚Äî without it trades can't read the projects row, so `projects!inner` embeds silently drop rows and `/projects/{id}/...` 404s.
- **Disclaimer** (0077): org-wide `app_settings` key `decision_disclaimer` (read-all-authenticated, staff write via `saveDecisionDisclaimer`). Edited in a box at the top of the decisions page; rendered at the bottom of every decision the CLIENT views (and in preview).
- **Client preview**: the drawer's "Preview as client" toggle (staff, edit mode) re-derives isClient/canEdit locally; approve/decline/request buttons render disabled. No impersonation ‚Äî same staff session, UI-only.
- **Catalog link** (0076): `decision_cost_items.catalog_item_id` (bare uuid ‚Äî cross-DB, no FK) + `catalog_item_code` snapshot point at `catalog_items` in the HH-SpecMagician Supabase project. `lib/supabase/specmagician.ts` (clone of the CRM client pattern; `SPECMAGICIAN_SUPABASE_URL`/`_SERVICE_ROLE_KEY`, null when unset) + `searchCatalogItems` in `app/actions/catalog.ts` power the per-line picker in both cost-item editors.
- **Export**: staff "Export" button on the decisions page builds a 3-sheet .xlsx (Decisions / Choices / Line items) client-side via the dependency-free writer `lib/export/xlsx.ts` (stored-zip + inline strings ‚Äî no npm dep).
- **Bulk copy**: staff multi-select decisions (checkbox column) ‚Üí floating bar ‚Üí `bulkCopyDecisions` (loops `copyDecision` per id: fresh draft, new number, approvals/choices reset, cross-project anchors dropped, assignments copied verbatim).

## Daily Logs module ‚Äî model

- `daily_logs(visibility = 'internal'|'client')` ‚Äî visibility is rendered prominently in the UI (left border + badge). `internal` is hidden from the future client portal; `client` is shown.
- Subs/vendors that were on site live in `daily_log_subs_on_site (daily_log_id, company_id, notes)`.
- Files in `daily_log_attachments` reference Supabase Storage objects in bucket `project-files`, with key `projects/{project_id}/daily-logs/{random}.{ext}`. Bucket is private. Server-side actions issue 1-hour signed URLs via `getSignedUrls` in `app/actions/daily-logs.ts`.

- **AI client-update composer**: "Draft client update" (staff, `daily-logs-client.tsx`) calls `draftClientUpdate` in `app/actions/daily-logs.ts`, which gathers the last week's INTERNAL logs plus recently-completed and upcoming work items and runs one Claude call (`lib/ai/client-update.ts`, single-shot like the insurance extractor) to rewrite them into a homeowner-friendly note. The draft prefills the daily-log drawer preset to `visibility = 'client'`; nothing publishes until the staffer saves via `saveDailyLog` (which fires the `daily_log.published` dashboard webhook).

## Schedule/To-Dos module ‚Äî model

- `schedule_items` is a single table; `kind` is `'work'` or `'todo'`. A to-do nests under a work item via `parent_id`. Standalone to-dos have `parent_id = null`.
- Every project carries two protected milestone work items (`schedule_items.milestone`: `'job_start'` | `'substantial_completion'`, unique per project). They move/complete/link like any work item but can never be deleted (DB trigger `protect_schedule_milestones` + app-layer guards). `ensureProjectMilestones` runs on every project-creation path and ADOPTS existing work items titled "Job Start"/"Substantial Completion" (case-insensitive ‚Äî the Template ships with them) before creating new rows (0070); the health banner offers a fallback button.
- **Baseline** (0069): "Set baseline" on the schedule page (`setScheduleBaseline` ‚Üí RPC `set_schedule_baseline`) copies each work item's current dates into `baseline_start_date`/`baseline_end_date` and stamps `projects.baseline_set_at`. Pre-baseline, work items can't be marked `complete` (to-dos can). Post-baseline, any date change to a work item requires a reason (`MoveReasonDialog` popup ‚Üí `schedule_delays` row); the AI apply path auto-logs under `other`. Work items created post-baseline get their initial dates stamped as baseline; `duplicateProject` copies milestones but nulls baselines. **Re-baseline is locked once any work item is complete**: staff can re-baseline (running `setScheduleBaseline` again) only while the job has zero completed work items, so a slipping schedule can't be re-baselined to look on-track; the first lock is always allowed. (Both the server action and the health banner's "Re-baseline" button enforce this.)
- **Health banner** (`components/schedule/health-banner.tsx`, math in `lib/schedule/health.ts`): job duration = Job Start ‚Üí Substantial Completion, vs baseline, with a 30-day buffer. Green "X Days Remaining in Buffer" (slip ‚â§ 30), yellow "X Days Late" (1‚Äì15 late), red (16+ late). While Substantial Completion is incomplete its projected date is clamped to today, so a stale schedule slips one day per day.
- Predecessors live in `schedule_predecessors`. FS is default. Cycle detection lives in `lib/schedule/scheduling.ts:wouldCreateCycle`.
- Cascading dates after a move: `cascadeFromPredecessors` returns successor updates; the server action applies them in a loop.
- **Recurring to-dos = roll-on-complete**: `recurrence_rule` jsonb ({freq daily|weekly|biweekly|monthly, interval, until, count, anchor_mode}) lives on exactly ONE row of a series. Completing that row (any path: checkbox, sheet, bulk, onsite quick-update, AI apply ‚Äî all call `lib/schedule/roll-recurrence.ts:rollRecurringTodo`) strips the rule off the completed row and inserts the next occurrence (checklist reset, assignments copied, count decremented). **`anchor_mode`** (absent = `fixed`): `fixed` keeps the cadence anchored to the due date (next due = smallest anchor-aligned slot strictly after max(due, completion) ‚Äî missed slots skip, "every May 1" stays May 1); `after_completion` re-anchors on the completion date (next due = completion + one interval ‚Äî the air-filters case). The Zod `Recurrence` schema in `app/actions/schedule.ts` must list every rule key or saves silently strip it. No rows are pre-created and nothing is virtual; `expandRecurrence` exists for future calendar use only (returns just the anchor row for after_completion ‚Äî future occurrences are unknowable). A rule without a due date is inert (the dialog blocks saving that). The onsite path passes `anchorDueOverride` because it rewrites due_date while completing, and `completionDateOverride` on "already done on‚Ä¶" so after_completion counts from the real work date.
- Priority flags: `PriorityBadge` renders ONLY for `high` (low/medium show nothing anywhere ‚Äî filter/sort still use all three).
- Assignments can target a profile (internal staff), a company (sub/vendor), or a role ‚Äî exactly one of the three must be non-null (0054). The bulk bar assigns/unassigns people AND roles (`bulkAssignRoleToScheduleItems` etc.).
- **Onsite quick updates can text the assignee**: `getOnsitePrompts` resolves each prompt's textable recipients (direct company assignment, or the company filling an assigned role via `project_role_members`); the card's "Text ‚Ä¶" panel prefills a per-trigger nudge and sends through the same `sendQuoTextToSub` action as the schedule dialog (server re-verifies the assignment).
- **Bulk copy to job**: the list view's bulk bar copies selected work items + to-dos to another project (`bulkCopyScheduleItems`): parent/child + predecessor edges survive when both ends are selected, statuses reset, checklists reset, assignments verbatim, milestones skipped, baseline stamped if the target is baselined.
- **Project History tab** (0073): `project_history` (staff-read, trigger-only writes via SECURITY DEFINER `record_project_history`) audits create/update/delete on schedule_items, decisions, daily_logs, project_files, project_payments, bid_packages, purchase_orders, project_members, project_role_members, schedule_assignments (+ projects updates) with per-field diffs and a `txid` batch key. `project_id` is a bare uuid (no FK) so project-delete cascades can't deadlock on it. Rendered at `/projects/[id]/history` (tab right of Roles, hidden from client/trade; latest 500, batched by txid).
- **Recently deleted / restore** (0088): `deleted_items` (staff-read, trigger-only writes, bare-uuid project_id like history) snapshots every DIRECT delete of schedule_items / decisions / daily_logs / project_files / bid_packages / purchase_orders ‚Äî full row + children (subtree to-dos, choices, line items, attachments, assignments, checklists, comments, predecessor edges) + link metadata (due-date anchors, follow-up materializations) + storage paths. Cascade deletes are skipped via `pg_trigger_depth() > 1` (the parent's snapshot covers them; project deletes leave no trash). The capture trigger is named `trg_a_trash_*` so it fires before `trg_si_decision_due_unlink` (BEFORE triggers run alphabetically). Delete actions no longer remove attachment Storage objects ‚Äî purge (0089/0090, `lib/trash-purge.ts`, run lazily on History-page load AND by the daily `/api/cron/trash-purge` sweep) is claim-batch (`purge_claimed_at` ‚Äî a claimed entry can't be restored, killing the check-vs-remove race; stale claims re-sweep after 1h) ‚Üí verify paths against live attachment tables (`unreferenced_storage_paths`) ‚Üí remove Storage objects ‚Üí finalize rows; a failed removal unclaims and retries next sweep, and 200-row batches loop until drained so PostgREST's 1,000-row cap can't truncate a backlog. Restore = `app/actions/trash.ts:restoreDeletedItem`: claim-then-insert via `claim_deleted_item` (idempotent; unclaimed on main-row failure), re-inserts under the caller's RLS session with fixups ‚Äî missing FK targets nulled/skipped with warnings, NOT-NULL `created_by` falls back to the restorer, reused decision/bid/PO numbers reallocated via the advisory-lock RPCs, predecessor edges cycle-checked (`wouldCreateCycle`), file version-chain head repaired, `claim_restored_entities` closes overlapping bulk-delete entries. UI: "Recently deleted" toggle on the History tab (`trash-panel.tsx`). Payments are excluded ‚Äî they keep their own soft-delete/restore (0022).

## Bid Requests module ‚Äî model

- `bid_packages` (per-project sequential `number`, statuses `draft ‚Üí sent ‚Üí awarded | closed`) with `bid_package_line_items` (cost-coded pricing structure, no unit cost) or `flat_fee` mode. Subs' pricing lives in `bid_line_item_quotes` / `bid_recipients.flat_total`.
- One `bid_recipients` row per invited company, with statuses `invited ‚Üí submitted | declined ‚Üí awarded` and an unguessable `token` ‚Äî the sub's only credential. Public page `/bid/{token}` (no login); mutations in `app/actions/bid-public.ts` run on the **admin client** with compare-and-swap status guards. **No anon RLS policies** ‚Äî the anon key can never touch bid/PO tables. Revocation = nulling the token (close/unrelease).
- Trade-role subs also see their own company's rows at `/my-bids` (RLS `br_trade_read` / `bp_trade_read`; never competitors' quotes). Cards link to the same token pages ‚Äî one response UI.
- Staff actions in `app/actions/bids.ts`. "Revise & re-request" (not silent edit) wipes quotes and resets non-declined recipients on a released package. `award_bid` RPC atomically awards + optionally creates a draft PO pre-filled from the winning quotes (`source_bid_recipient_id` links back).
- **Reopen** (`reopenBidPackage`): CAS `closed ‚Üí sent` + `closed_at=null`, then re-mints a fresh `generateAccessToken()` per recipient (one row at a time, `.is("token", null)` guard ‚Äî never restore revoked tokens) and re-notifies still-`invited` recipients. Quotes/statuses survive the close/reopen round trip; awarded packages are NOT reopenable.
- Send/notify via `sendEmail` + `sendQuoSms`, respecting `companies.notifications_enabled`. Sub submissions notify staff by email + in-app notification (inserted via admin client).

## Purchase Orders module ‚Äî model

- `purchase_orders` (per-project `number` + optional `custom_number`, statuses `draft ‚Üí released ‚Üí approved | declined`, plus `void`; `work_complete` is an independent flag) with `po_line_items` (cost-coded, with unit_cost) or flat fee. v1 has **no payments/bills** ‚Äî that stays in QuickBooks/Adaptive.
- Release mints a `token` and emails/SMSes the sub a public `/po/{token}` link; the sub approves with a typed signature + disclaimer checkbox (`approved_by_profile_id` null) or declines with a reason. Staff can approve on behalf (`staffApprovePurchaseOrder`, profile id recorded). Unrelease pulls it back to draft, revokes the token, and clears approval state; void keeps the record but kills the link.
- Structural edits are draft-only ‚Äî released POs must be unreleased first.
- Trade portal page `/my-pos` (RLS `po_trade_read`, own company + non-draft only).
- Approved POs roll up as **Committed costs** by cost code in the Budget tab's POs column ‚Äî staff with `profiles.financial_access` only, never clients. (The old Pricing-tab Committed costs card was removed; Pricing shows only the client-facing contract picture.)
- Numbering RPCs `next_bid_package_number` / `next_po_number` use per-project advisory locks (hash args 1 / 2; decisions use 0). `award_bid` allocates PO numbers under the same lock key as `next_po_number`.
- **PO provenance**: `source_bid_recipient_id` (award path or `createPoForBidRecipient` ‚Äî the "sub never responded, moving ahead anyway" thin action: prefills from the package, prices from the sub's quotes when present else 0, touches neither package nor recipient status) and `source_decision_id` (0095; `createPoFromDecision` on an APPROVED decision ‚Äî selection copies the approved choice's `decision_cost_items`, change order the decision-level rows, at **raw unit_cost, never markup**; duplicate POs from one decision are a soft warn via `already_linked`). `source_decision_id` is the only purchase_orders‚Üîdecisions FK ‚Äî resolve chips with **separate queries, never embeds** (PGRST201). Decision drawer's approved footer has "Create PO‚Ä¶" (inline vendor picker, preselects a lone company assignment).

## Budget module ‚Äî model

- **Per-project budget by cost code** (0087): `project_budget_lines` (unique project+cost_code; `budget_amount` + nullable `forecast_override`) and `project_cost_actuals` (unique project+cost_code; `amount`, `source` import|manual|qbo, `as_of`) ‚Äî QBO will upsert actuals per cost code once Hines Homes is on QuickBooks Online; until then the spreadsheet import's optional "Actual costs" column stages them. Both tables staff-only RLS; **the whole Budget tab additionally requires `profiles.financial_access`** (app-layer, like committed costs) ‚Äî page, server actions (`app/actions/budget.ts:requireFinancialStaff`), and nav (`SectionTabs` gets a `financialAccess` prop; sections can set `requiresFinancialAccess`). **Writes are further gated by the budget-editors allowlist** (`app_settings` key `budget_editors`, picked at `/settings/budget` via the avatar menu): never-set = every financial_access staffer edits (pre-allowlist behavior), explicit list = only those ids, `[]` = read-only for all; a read failure or malformed row **fails closed** to read-only. Only CURRENT editors can change a non-empty list (`saveBudgetEditors` self-guard; unset/empty stays fixable by any financial_access staffer so there's no lockout). `requireBudgetEditor` wraps every mutation (including `parseBudgetImport`, so readers don't get a preview that dead-ends); the page passes `canEdit` and non-editors see plain display cells (amounts still visible ‚Äî only editing is gated). Same app-layer trust tier as financial_access (any staff can edit app_settings) ‚Äî documented, not RLS.
- **Derived columns** (math in `lib/budget/rollup.ts`, UI-free): Changes to Budget allocates each APPROVED decision's `cost_delta` to the cost codes on its line items (chosen choice's items for selections, marked up by `markup_percent`; allowance credits `allowance_cost_code_id`; delay cost / manual deltas / rounding fall to the "Uncoded / flat fee" bucket ‚Äî so the column total always equals the Pricing tab's Approved changes). New Budget = Budget + Changes. POs column = approved-PO committed costs (flat-fee POs ‚Üí uncoded, same math as the Pricing card). Forecasted Remaining = `forecast_override ?? (New Budget ‚àí Actuals)` (click cell to override, clear to revert; overrides render amber with `*`). Total Forecasted = Actuals + Forecasted Remaining; Variance = Total Forecasted ‚àí New Budget (positive red = over).
- **Spreadsheet import**: `lib/import/spreadsheet.ts` is a dependency-free server-side .xlsx/.csv READER (zip + `node:zlib` inflate, shared/inline strings; first worksheet only) ‚Äî the read counterpart of `lib/export/xlsx.ts`. Flow: `parseBudgetImport(FormData)` returns a preview (rows matched by the cost-code column, skipped rows with reasons), then `applyBudgetImport` merge-upserts. Template + table export buttons build .xlsx client-side via `makeXlsx`. Budgets will eventually transfer from the SpecMagician bid tool once it's finished; the importer stays for QBO-era corrections.

## Client invoices module ‚Äî model (QBO hybrid)

- **QuickBooks stays the invoicing system of record**: invoices are created in QBO, QBO emails the client + sends reminders + takes payment on Intuit's hosted pay page (and emails the receipt). The app only MIRRORS them into the portal ‚Äî no invoice creation, no payment processing, no client notifications from our side.
- Link (0087): `projects.qbo_customer_id` + `qbo_customer_name` snapshot point a job at its QBO Customer. Staff link/change/unlink in the Invoices tab dialog (`linkProjectQboCustomer` purges the project's cached rows then backfills; unlink hard-deletes them ‚Äî a wrong-customer link must never leave another client's invoices visible).
- Cache: `qbo_invoices` (unique per realm+QBO id, bare-uuid `project_id` like qbo_po_sync). **Service-role writes only**; staff read everything, clients read `open`/`paid` on their `project_members` jobs, trades nothing. `status`: open | paid (balance 0) | voided | deleted (tombstone for rows QBO no longer returns); "Overdue" is derived in the UI from due_date + balance, not stored.
- Sync paths in `lib/quickbooks/invoices.ts` (reuses the 0085 OAuth connection): `syncProjectInvoicesFromQbo` full-reconciles one customer (link-time backfill + staff "Sync now"), `syncSingleInvoice` handles one id (webhook). `include=invoiceLink` rides on reads/queries for the hosted pay link ‚Äî null when the QBO company hasn't enabled online payments (staff see "No pay link", clients see nothing).
- Webhook `/api/qbo/webhook`: configured in the Intuit dev portal (Invoice + Payment entities, all ops) with `QBO_WEBHOOK_VERIFIER_TOKEN` (HMAC-SHA256 verify). ACKs immediately, processes in `after()`. Payments resolve to invoices via the Payment's LinkedTxn. Deliveries can duplicate/reorder ‚Äî everything is an idempotent upsert, and the staff in-app notification (`invoice_payment`) fires only on an observed balance DROP, so replays stay silent. New rows never notify (that's backfill). Recipients come from `app_settings` key `invoice_payment_recipients` (JSON profile-id array, picker in Settings ‚Üí QuickBooks; currently Brandon + Megan) ‚Äî never-set key falls back to `financial_access` staff, explicit empty list = notify nobody.
- Settings ‚Üí QuickBooks has the webhook setup checklist (endpoint URL, entities, verifier-token env var).

## Vendor Documents module (subcontractor insurance / W9 / SMA) ‚Äî model

- `insurance_documents` (one per ingested file) + `insurance_policies` (one per policy parsed off it; enum `insurance_type`: general_liability | workers_comp | auto | umbrella). "Current" policy for a company+type = latest `expiration_date`; older rows are history ‚Äî so GL and WC arriving on two separate certs with different dates just works. Staff-only RLS on both.
- **Doc kinds** (0091, widened 0097): `insurance_documents.doc_kind` = `coi` (default) | `w9` | `sma` | `other`. Every ingest path AUTO-CLASSIFIES by default: the Claude extraction returns `doc_kind` and the pipeline files W9s/SMAs to their auto-matched company exactly like certificates (email + sub-token uploads included ‚Äî no longer COI-only). `other` = the classifier couldn't recognize the document ‚Üí always `needs_review` (the review card has a kind select so staff correct the classification and assign in one step; `assignInsuranceDocument` takes the optional kind and only materializes policies / fills agent contact when the FINAL kind is `coi`). Staff can still pick an explicit type in the upload dialog (select defaults to Auto-detect; company is always optional now); explicit W9/SMA + company skips the model call entirely, and an extraction failure on an explicit W9/SMA degrades to `needs_review`, not `failed`. The dashboard shows a per-company Docs column (latest W9/SMA chips) and lists W9/SMA/other docs in the expanded row.
- Three ingestion paths, all funneling into `lib/insurance/ingest.ts` (admin client): the Resend inbound webhook `/api/inbound/insurance` (verify with `RESEND_WEBHOOK_SECRET`, download attachments via `resend.emails.receiving.attachments`), the public tokenized sub upload (`/insurance-upload/{companies.insurance_upload_token}` ‚Üí POST `/api/insurance-upload`), and staff manual upload (browser ‚Üí Storage, then `processStoredInsuranceDocument`; multi-file, with page-wide drag-and-drop that preloads the dialog).
- Extraction: `lib/insurance/extract.ts:extractVendorDocument` reads the PDF/image with Claude (`claude-opus-4-8`, structured outputs; single-coverage certs are explicitly fine) and returns `doc_kind` + `company_name` (insured for COIs, line-2/line-1 business name for W9s, subcontractor party for SMAs) + policies (COI-only). Result is stored on `insurance_documents.extraction` so the review queue can assign an unmatched doc without re-running the model (old stored extractions predate `doc_kind` ‚Äî readers fall back to the row's column). It also captures the ACORD **Producer** (agency name/email/phone), which `fillAgentContactFromExtraction` writes onto the company's BLANK `insurance_agent_*` columns (staff values never overwritten; also runs on manual assign of a COI).
- Company auto-match (`matchCompany`, all doc kinds): sender email exact match ‚Üí **document history** (a past `processed` doc whose normalized extracted name matches teaches the mapping ‚Äî staff assigning a doc once in the review queue fixes that spelling forever; all history must agree on one company; processed W9s/SMAs feed the same history) ‚Üí unambiguous name equality against `companies.name` OR `companies.aka` (suffix/punctuation-insensitive, `&`‚â°`and`; never a bare contains-match) ‚Üí **token-subset** (every word of the shorter name in the longer one ‚Äî "Affordable Gutters" ‚äÇ "Affordable Gutters Plus Siding and Insulation LLC"; shorter side needs ‚â•2 words, exactly ONE directory match). Anything fuzzy/ambiguous ‚Üí `status='needs_review'`. **Batch assign**: `assignInsuranceDocument` also files every other queued (non-`other`) doc whose extracted name normalizes to the same string (returns `alsoFiled` for the toast) ‚Äî assigning one of a sub's three certs files all three.
- **AKA** (0091): `companies.aka` ‚Äî `name` is the official name (payments/insurance), `aka` the everyday one. Edited in the company dialog, shown + searched on Companies and Insurance pages, and consulted by COI matching.
- **Insurance agent contact** (0091): `companies.insurance_agent_name/email/phone`, edited in the company dialog and auto-filled from cert Producers. `sendInsuranceRequest` and the reminder cron **CC the agent email** (agent becomes To when the company has no email; "no email" now means neither).
- **Bulk audit export**: checkbox multi-select on the dashboard ‚Üí POST `/api/insurance/export` (staff-gated route; caller's RLS session reads docs + Storage) ‚Üí one ZIP, folder per company, containing the docs behind each company's current policies + latest W9/SMA. The selection bar has per-type include checkboxes (Certificates / W9s / SMAs) that filter which doc ids are sent. ZIP writer is `lib/export/zip.ts` (STORED entries, factored out of `lib/export/xlsx.ts`).
- Reminders: daily cron `/api/cron/insurance-reminders` (auth `CRON_SECRET`) emails each company once per policy when a CURRENT policy expires within 7 days, with their upload link, then stamps `reminder_sent_at`. Respects `companies.notifications_enabled`; the staff "Send request" button does not (explicit click wins). Gated by a global kill switch: the cron no-ops unless `INSURANCE_REMINDERS_ENABLED === "true"` (OFF by default so automatic emails don't go out before the site is live; the manual button is never gated).
- Only companies with `companies.status = 'Approved for Use'` (case-insensitive; helper `lib/insurance/requirements.ts:companyRequiresInsurance`) are REQUIRED to carry insurance ‚Äî the cron skips everyone else and the dashboard neither lists them by default nor flags their missing coverage ("Show all statuses" reveals them).
- Files live in the `project-files` bucket under `companies/insurance/‚Ä¶` (staff storage RLS already covers it; server paths use the admin client).
- Staff UI: `/companies/vendor-documents` ("Vendor Documents", in the People menu right under Companies; `/companies/insurance` is a redirect) ‚Äî coverage table (GL/WC required; auto/umbrella tracked), review queue, manual upload, send-request. Storage keys keep the `companies/insurance/‚Ä¶` prefix and the sub-facing upload page stays at `/insurance-upload/{token}`.

## AI smart-update agent ‚Äî model

- Server action `runAgentTurnAction` in `app/actions/ai-agent.ts` wraps a manual Claude tool-use loop in `lib/ai/agent.ts`. Model is `claude-sonnet-5` with adaptive thinking on (a tool loop needs the model to keep reaching for tools) and `max_tokens` 8192 so thinking + tool calls + summary don't truncate. The tools + system prefix is prompt-cached (a `cache_control` breakpoint on the system block, stable within a turn). Requires `ANTHROPIC_API_KEY` env var (set in Vercel + `.env.local` for dev) ‚Äî action returns a typed `error` result if the key is missing, never throws.
- Plan-then-approve flow: the agent's `propose_*` tools record mutations into a per-turn array but DON'T execute anything. Only `applyPlanAction` actually writes to the DB, and it runs under the caller's session so RLS still gates writes. Each plan carries a server-generated `plan_id`; `applyPlanAction` claims it in `ai_plan_applications` (unique) BEFORE applying, so a re-apply (double-click, or the walkthrough's blind retry) returns the first apply's stored results instead of duplicating writes / re-texting subs. That table is also the audit trail (proposed `mutations` + `results` per apply). If a turn is cut short (`stop_reason` max_tokens, or the iteration cap), the plan result carries `incomplete` and `PlanCard` shows an amber warning.
- Adding a new mutation kind takes five changes: extend `ProposedMutation` in `lib/ai/types.ts`, add the `propose_*` tool definition + handler in `lib/ai/agent.ts`, add the apply path in `lib/ai/apply.ts`, add a case in the shared plan-row renderer `components/ai/plan-review.tsx:MutationRow`, and extend the Zod `MutationSchema` in `app/actions/ai-agent.ts`. Mutation kinds: `add_checklist_item`, `create_todo` (optional assignee), `create_work_item`, `create_decision`, `add_decision_followup`, `append_daily_log` (optional photos + `subs_on_site`), `update_schedule_item_status`, `update_schedule_item`, `update_decision_status`, `assign_schedule_item`, `send_sms`, `send_bid_reminder`. Read-only tools also answer questions (reporting mode): `list_staff`, `get_schedule_health`, `list_schedule_delays`, `list_daily_logs`, `list_bid_packages`/`get_bid_package`, `list_purchase_orders` (status only ‚Äî never dollar amounts; PO costs are `financial_access`-gated at the app layer, not RLS).
- **Help-desk mode**: the agent also explains how the APP itself works ("how do I set a baseline?", "what's a change order vs a selection?"). Knowledge lives in `lib/ai/help-content.ts` as staff-facing `HELP_TOPICS` (one per module ‚Äî product docs, no RLS/migration jargon). `helpCatalog()` renders the id+title+summary of every topic into the (cached) system prompt so the model always sees the full menu; the read-only `search_help_topics` tool (`{ query?, topic_ids? }`, ranked substring match in `searchHelpTopics`) returns the full `body` of matched topics on demand. Adding/updating a topic is a one-file edit ‚Äî keep bodies in sync with the real UI. Help answers propose nothing and never append a daily log (same guard as reporting mode). The system prompt's "three hats" intro routes a message to help-desk / reporting / field-notes by intent.
- Field-notes mode: dictated/typed site notes map to schedule updates, to-dos (optionally assigned to staff via `list_staff` or a sub/vendor), assignments on existing items (`assign_schedule_item`, in-app notification to a staff assignee), an SMS to a sub (`send_sms`, sent via `lib/quo.ts` with the recipient re-resolved from `companies.phone` at apply time), a bid reminder to invited-but-unresponded recipients (`send_bid_reminder`, reuses each recipient's existing token), a **draft change order** when the note describes the homeowner asking for extra/changed work (explicit prompt mapping ‚Üí `create_decision` kind `change_order`; stays draft for staff to price), and ALWAYS one `append_daily_log` per affected project (appends to, or creates, the internal daily log for that date; records any on-site subs in `daily_log_subs_on_site`). The browser sends its local date as `today` so "today" means the user's day, not UTC's. The chat dialog has a Web Speech API mic button for phone dictation (shims shared via `lib/speech/web-speech.ts`), and auto-scopes to the project when opened from a `/projects/[id]` route (`context.mode: "page"` ‚Äî a dismissible "Scoped to ‚Ä¶" chip; no photos, unlike the onsite `mode`).
- Onsite walkthrough (`components/onsite/walkthrough.tsx`, top of the project Onsite tab): voice-memo segments + photos feed the same agent with a trusted `context: { project_id, attachments }` on `runAgentTurnAction`. The server re-resolves the project under the caller's RLS session and adds an "On-site context" section to the system prompt (explicit carve-out to the IDs-from-tool-results rule for that one server-verified id). Photos never go to the model ‚Äî after the plan returns, the action injects them into its `append_daily_log` for that project (synthesizing one if missing) so they deterministically land as `daily_log_attachments`; apply rejects storage paths outside `projects/{project_id}/daily-logs/`. Review is per-item checkboxes + a two-tap confirm for destructive rows (the global dialog keeps its all-or-nothing typed-"apply" gate). Drafts (segments + uploaded photo paths) persist in localStorage per project; photos orphaned by a discarded draft or an unchecked log row are accepted as unreferenced private-bucket objects (v1 policy).

## Communications / Quo phone ‚Äî model

- Every outbound text/call and inbound event is logged to `communications` (see the Daily Logs/Bids modules for feed rendering). `sendQuoSms` in `lib/quo.ts` is the single SMS sender; `app/api/inbound/quo/route.ts` ingests the OpenPhone webhook (`message.received/.delivered`, `call.completed`, `call.recording.completed`).
- **Per-user Quo numbers** (0081): staff can be assigned their own Quo (OpenPhone) number on the Team page ‚Äî stored as `profiles.quo_phone_number_id` ("PN‚Ä¶", the API `from`) + `profiles.quo_phone_number` (E.164), 1:1 via partial-unique indexes. `sendQuoSms` resolves the sending number from `senderProfileId ?? log.sent_by` (which every call site already sets to the acting staffer) ‚Üí their stored number ‚Üí the shared `QUO_FROM_NUMBER` fallback. So automated texts (bid/PO/assignment/manual/hub-reply/AI-apply) go out from the initiator's own number with **zero call-site changes** ‚Äî they all pass `log.sent_by`. An explicit `opts.from` still overrides.
- The Team picker is populated by `listQuoPhoneNumbers()` (`GET /v1/phone-numbers`); empty when `QUO_API_KEY` is unset, and the dialog then only offers the shared number. `updateProfile` maps a 23505 to a friendly "already assigned" error; a non-staff role clears the assignment.
- **Native attribution**: the webhook matches each event's business-side number (`phoneNumberId` first, then E.164) to a profile via `staffProfileForQuoNumber`. Outbound rows get `sent_by` = that staffer (so texts they type and calls they dial in the Quo app show their name in the feed, which renders `sent_by` for outbound); inbound rows keep the counterparty as author but stash the line owner in `meta.quoStaffProfileId`. App-sent texts already hold `(source,provider_id)` with richer attribution, so the webhook's `ON CONFLICT DO NOTHING` leaves them untouched. Note: the Quo/OpenPhone API can only *send texts* and *track* calls (via webhook) ‚Äî it cannot *place* calls.
- **Compose** ("New message"): staff can START a text or email from either communications tab ‚Äî `ComposeMessageButton` (`components/comms/compose-dialog.tsx`) ‚Üí `composeMessage` in `app/actions/communications.ts`. Recipients resolve server-side (a company from the directory, or the job's client contact slots on the project tab; a `custom` raw number/address is the deliberate escape hatch). SMS rides `sendQuoSms` (staffer's own Quo number), email rides `sendEmail` (Outlook‚ÜíResend); both log as `manual_sms`/`manual_email` with company/project attribution ‚Äî plus the client's portal profile (best-effort `project_members` match) so a client-directed message shows in their portal feed too. Project-tab sends file to that job; global-hub sends stay global unless the recipient is a job's client contact. Company contact lists come from `lib/comms/contacts.ts:buildCompanyContacts`. **Clients** get their own `ClientComposeButton` on the project tab (no recipient picker ‚Äî always "the team"): `clientComposeMessage` verifies membership via the RLS-scoped project read, inserts the inbound `communications` row via the admin client (`source_kind='client_message'`, stamped with their profile+project so both sides' feeds show it), then fans out staff in-app notifications + email (the requestDueDateReset pattern).
- **Job attribution & the global hub**: `matchCounterparty` (`lib/comms/match.ts`, via the `match_contacts_by_phone/email` service-role RPCs in 0067) resolves an inbound number/address to a project when it can (project client contact ‚Üí profile ‚Üí **recent conversation** ‚Üí company's single active engagement). The recency heuristic (`recentProjectForCompany`, phone traffic only) files a sub's reply to the job we last texted them about: all project-stamped rows for that company+number in the last 7 days must agree on ONE project, and the evidence must include an outbound/staff row (an auto-filed inbound can't feed itself). The hub offers "Re-file to a different job" on filed rows as the correction path; AI-sent texts stamp project_id when the company's schedule assignments are single-project. Quo phone traffic is **never queued for manual placement**: the webhook always writes `status='logged'`, so it auto-files to a job only when the matcher is confident (a single-job sub) and otherwise stays global-only. A conversation can span multiple jobs, so this is intentional ‚Äî staff don't have to file anything. The global staff hub `/communications` (`GlobalCommunicationsClient`) is the searchable home for all channels (shows everything except `status='ignored'`); unfiled rows offer a quiet optional "File to job" (`assignCommunication`) / "Dismiss" (`ignoreCommunication`) ‚Äî there is no nagging needs-review queue. Job-level `/projects/[id]/communications` still shows only `project_id=this AND status='logged'`. (Inbound email uses the same matcher but prefers the `+p_<project_id>` reply plus-tag; unmatched email keeps `status='needs_review'`, which now renders identically to any other unfiled row in the hub.)
