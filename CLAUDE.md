# Project Manager — agent notes

## Conventions

- **Next.js 16 App Router** with Turbopack (default). `middleware.ts` is renamed to `proxy.ts` (default export named `proxy`).
- **Async APIs**: `cookies()`, `headers()`, `params`, `searchParams` are all Promises. Always `await` them.
- **Supabase SSR** via `@supabase/ssr`. Server client in `lib/supabase/server.ts`; browser client in `lib/supabase/client.ts`; session refresh in `lib/supabase/proxy.ts` (invoked from `proxy.ts`).
- **Auth helpers**: `getSessionProfile`, `requireSession`, `requireStaff` in `lib/auth.ts`. Use these in server components & actions instead of querying `auth.users` directly.
- **Server actions** live in `app/actions/`. Validate input with `zod`. Use `revalidatePath` after mutations.
- **RLS** is the source of truth for access control. Don't hand-roll role checks in queries — RLS rejects rows the user can't see. The `requireStaff` helper is just a UX redirect.
- **UI primitives** in `components/ui/` (Button, Input, Dialog, Card, Badge, Avatar, EmptyState). Minimal — extend rather than reach for shadcn unless the need is repeated.
- **Design tokens** in `app/globals.css` under `@theme`. Use `bg-brand-500`, `text-foreground`, `border-border`, etc. instead of raw Tailwind colors so the look stays consistent.

## Database

- Migrations: `supabase/migrations/NNNN_name.sql`.
- Apply via Supabase MCP `apply_migration` or `supabase db push`.
- Regenerate types: `supabase gen types typescript --project-id <ref> > lib/db/types.ts`.
- After any DDL, run `get_advisors` (security + performance) and fix WARNs.

## Projects & templates — model

- **New project dates**: `projects.start_date` = the CRM's "Projected Start Date" (pulled through the dashboard; `normalizeDashboardProject` accepts `start_date`/`projected_start_date`/`projected_start`). On the template-copy path (`duplicateProject`), the schedule is anchored on the **Job Start milestone** so it lands exactly on `start_date` (falls back to the earliest dated item if the template has no dated Job Start); every item shifts by the same delta. Blank projects stamp their Job Start milestone with `start_date` too. There is **no** `target_completion_date` — the column was dropped (0081); a job's projected finish is the Substantial Completion milestone + health banner, not a project field.
- **Role assignments carry over from the template**: `duplicateProject` copies the source's `project_role_members` (the "who fills each role" map — Project Manager, Site Superintendent, every trade role) verbatim into the new job. Roles are org-wide, so `role_id` copies directly; only `project_id` changes and the profile/company assignee is copied as-is. (Role-*based schedule assignments* already carried forward separately.) So every job built from the Template inherits the same people/companies in the same roles.
- **Purchasing carries over from the template**: `duplicateProject` also copies purchase orders and bid packages as **fresh drafts** — numbers re-allocated 1..N in source order, all workflow state reset (tokens/release/approval/awards cleared, `work_complete` off), line items + vendor kept. `void` POs are skipped; a PO's `source_decision_id` remaps through the cloned decisions while `source_bid_recipient_id` drops (the copied bid round never happened on the new job). Bid recipients copy as the invite list (`invited`, no token — tokens mint at Send); quotes never copy. Attachments blob-copy like decision attachments (failures skip, counted in `purchasingAttachmentsFailed`, surfaced in the duplicate toast); 0095 Files-tab links (`project_file_id`) drop because `project_files` aren't cloned. Dates (`approval_deadline`, bid `due_date`) shift with the schedule. Any child-insert failure mid-clone deletes the new project shell (children cascade) before rethrowing, so the project number can be retried.
- **Status mirrors the CRM at creation**: both the blank/dashboard create path (`createProject`) and every template-copy path (`duplicateProject`) resolve `status` from the CRM by `project_number` via `getCrmProjectStatus` (`lib/supabase/crm.ts`, same `crmStatusToEnum` mapping as `syncProjectsFromCrm`) and store the verbatim `crm_status` word. So a new job matches the dashboard from birth instead of defaulting to "In Work" until the next "Sync from CRM". The CRM wins over the form's status / `override_status`; falls back to the form value when the CRM isn't configured or has no matching row.
- **Smart templates** (0042): template `schedule_items`/`decisions` carry `template_tags text[]`; `projects.attributes jsonb` stores the boolean answers captured at creation. An item copies only when `matchesTemplateTags` — every tag matches (`walkout` needs true, `!walkout` needs false; empty = always copy). Helpers in `lib/template-tags.ts`. Template tags are surfaced as purple chips (`components/template-tag-badges.tsx`) on the schedule list view AND the decisions table **only when `projects.is_template`** — inert and hidden once copied into a real job. Both editors (schedule dialog + decision drawer) expose the tags field to staff everywhere.
- **Either/or tag groups**: staff group tags into mutually-exclusive choices in Settings → Template tags (avatar menu, staff-only, `/settings/template-tags`), stored as JSON in `app_settings` key `template_tag_groups` (`getTemplateTagConfig`/`saveTemplateTagConfig` in `app/actions/settings.ts`). A **required** group renders as a single-select in the create/duplicate questionnaire (`TemplateOptionsFields`) and blocks "Create project"/"Duplicate" until answered (`TemplateOptionsValue.valid`); picking an option sets that tag true and its siblings false via `attributesWithGroupSelections`, riding on the same boolean-attributes matcher. Enforcement is client-side (staff-only flows).

## Decisions module — model

- One page for both `change_order` and `selection` kinds — they share table `decisions` with a `kind` enum and a per-project sequential `number`.
- Workflow: `draft` → `pending_client` → `approved` | `rejected`. Clients can only see decisions once they leave `draft`.
- `decision_followup_templates` are per-decision to-do templates the user defines while drafting. On the `approved` transition, `materializeFollowups` in `app/actions/decisions.ts` creates real `schedule_items (kind='todo')` with `source_decision_id` set, plus assignments and in-app notifications for staff assignees. Re-approval is idempotent via the `decision_followup_materializations` junction (PK decision_id+template_id, claim-then-insert) — NOT title matching.
- Due dates can be **linked to a schedule item** (0072): `due_anchor_schedule_item_id` + `due_anchor` (start|end) + `due_anchor_offset_days` on `decisions`, all-or-nothing triple. `due_date` stays canonical — `saveDecision` computes it from the anchor on save, and the `schedule_items` trigger `sync_decision_due_anchors` refreshes it on any date move (covers cascades, bulk shifts, AI apply) and unlinks on delete, freezing the last date. The same trigger clears follow-up-template anchors on delete (the 0035 SET NULL + triple-check combo used to make those deletes fail). Cross-project copies drop the link; `duplicateProject` remaps it through the cloned schedule.
- **PGRST201 warning**: decisions↔decision_choices (via `decision_id` + `selected_choice_id`) and decisions↔schedule_items (via `due_anchor_schedule_item_id` + `source_decision_id`) each have TWO relationships — always FK-hint embeds between these pairs (e.g. `decision_choices!decision_choices_decision_id_fkey(...)`), or PostgREST 300s and the embed comes back empty.
- `decision_comments`: staff full access; clients in `project_members` for the project can both READ and INSERT (RLS enforces `author_id = auth.uid()`). Trades have no access.
- Decision attachments share the `project-files` Storage bucket with daily logs. The storage RLS policy was extended in 0004 to allow client read for both daily-log AND decision attachments via signed URLs.
- **Past-due gate** (0074): the `client_decide_decision` RPC rejects APPROVE when `due_date < current_date` (decline stays allowed). The drawer's client footer disables Approve and offers "Request due-date reset" → `requestDueDateReset` (client-callable; leaves a decision_comment + notifies staff in-app/email via the admin client). Staff reset = just edit the due date.
- **Decision assignments** (0075): `decision_assignments` mirrors schedule_assignments (exactly one of profile/company/role, unique nulls-not-distinct). Staff assign selections in the drawer; `saveDecision` wipe-and-reinserts from `input.assignments`. Trades gain READ on assigned non-draft decisions + their choices/attachments (helper `trade_sees_decision`, storage policy extended); surfaced on `/my-assignments` under "Selections". Cost items stay staff-only. 0078 adds `projects_trade_read` (helper `trade_sees_project`: schedule assignment, role membership, or decision assignment) — without it trades can't read the projects row, so `projects!inner` embeds silently drop rows and `/projects/{id}/...` 404s.
- **Disclaimer** (0077): org-wide `app_settings` key `decision_disclaimer` (read-all-authenticated, staff write via `saveDecisionDisclaimer`). Edited in a box at the top of the decisions page; rendered at the bottom of every decision the CLIENT views (and in preview).
- **Client preview**: the drawer's "Preview as client" toggle (staff, edit mode) re-derives isClient/canEdit locally; approve/decline/request buttons render disabled. No impersonation — same staff session, UI-only.
- **Catalog link** (0076): `decision_cost_items.catalog_item_id` (bare uuid — cross-DB, no FK) + `catalog_item_code` snapshot point at `catalog_items` in the HH-SpecMagician Supabase project. `lib/supabase/specmagician.ts` (clone of the CRM client pattern; `SPECMAGICIAN_SUPABASE_URL`/`_SERVICE_ROLE_KEY`, null when unset) + `searchCatalogItems` in `app/actions/catalog.ts` power the per-line picker in both cost-item editors.
- **Export**: staff "Export" button on the decisions page builds a 3-sheet .xlsx (Decisions / Choices / Line items) client-side via the dependency-free writer `lib/export/xlsx.ts` (stored-zip + inline strings — no npm dep).
- **Bulk copy**: staff multi-select decisions (checkbox column) → floating bar → `bulkCopyDecisions` (loops `copyDecision` per id: fresh draft, new number, approvals/choices reset, cross-project anchors dropped, assignments copied verbatim).

## Daily Logs module — model

- `daily_logs(visibility = 'internal'|'client')` — visibility is rendered prominently in the UI (left border + badge). `internal` is hidden from the future client portal; `client` is shown.
- Subs/vendors that were on site live in `daily_log_subs_on_site (daily_log_id, company_id, notes)`.
- Files in `daily_log_attachments` reference Supabase Storage objects in bucket `project-files`, with key `projects/{project_id}/daily-logs/{random}.{ext}`. Bucket is private. Server-side actions issue 1-hour signed URLs via `getSignedUrls` in `app/actions/daily-logs.ts`.
- Browser uploads go directly to Storage with the user's JWT (RLS policy `project_files_staff_all`). The action `saveDailyLog` then records the path in `daily_log_attachments`.
- Clients can `select` from `daily_logs` only when `visibility = 'client'` AND they're in `project_members` for that project — enforced by RLS. Trades have no access to daily logs.
- **AI client-update composer**: "Draft client update" (staff, `daily-logs-client.tsx`) calls `draftClientUpdate` in `app/actions/daily-logs.ts`, which gathers the last week's INTERNAL logs plus recently-completed and upcoming work items and runs one Claude call (`lib/ai/client-update.ts`, single-shot like the insurance extractor) to rewrite them into a homeowner-friendly note. The draft prefills the daily-log drawer preset to `visibility = 'client'`; nothing publishes until the staffer saves via `saveDailyLog` (which fires the `daily_log.published` dashboard webhook).

## Schedule/To-Dos module — model

- `schedule_items` is a single table; `kind` is `'work'` or `'todo'`. A to-do nests under a work item via `parent_id`. Standalone to-dos have `parent_id = null`.
- Every project carries two protected milestone work items (`schedule_items.milestone`: `'job_start'` | `'substantial_completion'`, unique per project). They move/complete/link like any work item but can never be deleted (DB trigger `protect_schedule_milestones` + app-layer guards). `ensureProjectMilestones` runs on every project-creation path and ADOPTS existing work items titled "Job Start"/"Substantial Completion" (case-insensitive — the Template ships with them) before creating new rows (0070); the health banner offers a fallback button.
- **Baseline** (0069): "Set baseline" on the schedule page (`setScheduleBaseline` → RPC `set_schedule_baseline`) copies each work item's current dates into `baseline_start_date`/`baseline_end_date` and stamps `projects.baseline_set_at`. Pre-baseline, work items can't be marked `complete` (to-dos can). Post-baseline, any date change to a work item requires a reason (`MoveReasonDialog` popup → `schedule_delays` row); the AI apply path auto-logs under `other`. Work items created post-baseline get their initial dates stamped as baseline; `duplicateProject` copies milestones but nulls baselines. **Re-baseline is locked once any work item is complete**: staff can re-baseline (running `setScheduleBaseline` again) only while the job has zero completed work items, so a slipping schedule can't be re-baselined to look on-track; the first lock is always allowed. (Both the server action and the health banner's "Re-baseline" button enforce this.)
- **Health banner** (`components/schedule/health-banner.tsx`, math in `lib/schedule/health.ts`): job duration = Job Start → Substantial Completion, vs baseline, with a 30-day buffer. Green "X Days Remaining in Buffer" (slip ≤ 30), yellow "X Days Late" (1–15 late), red (16+ late). While Substantial Completion is incomplete its projected date is clamped to today, so a stale schedule slips one day per day.
- Predecessors live in `schedule_predecessors`. FS is default. Cycle detection lives in `lib/schedule/scheduling.ts:wouldCreateCycle`.
- Cascading dates after a move: `cascadeFromPredecessors` returns successor updates; the server action applies them in a loop.
- **Recurring to-dos = roll-on-complete**: `recurrence_rule` jsonb ({freq daily|weekly|biweekly|monthly, interval, until, count, anchor_mode}) lives on exactly ONE row of a series. Completing that row (any path: checkbox, sheet, bulk, onsite quick-update, AI apply — all call `lib/schedule/roll-recurrence.ts:rollRecurringTodo`) strips the rule off the completed row and inserts the next occurrence (checklist reset, assignments copied, count decremented). **`anchor_mode`** (absent = `fixed`): `fixed` keeps the cadence anchored to the due date (next due = smallest anchor-aligned slot strictly after max(due, completion) — missed slots skip, "every May 1" stays May 1); `after_completion` re-anchors on the completion date (next due = completion + one interval — the air-filters case). The Zod `Recurrence` schema in `app/actions/schedule.ts` must list every rule key or saves silently strip it. No rows are pre-created and nothing is virtual; `expandRecurrence` exists for future calendar use only (returns just the anchor row for after_completion — future occurrences are unknowable). A rule without a due date is inert (the dialog blocks saving that). The onsite path passes `anchorDueOverride` because it rewrites due_date while completing, and `completionDateOverride` on "already done on…" so after_completion counts from the real work date.
- Priority flags: `PriorityBadge` renders ONLY for `high` (low/medium show nothing anywhere — filter/sort still use all three).
- Assignments can target a profile (internal staff), a company (sub/vendor), or a role — exactly one of the three must be non-null (0054). The bulk bar assigns/unassigns people AND roles (`bulkAssignRoleToScheduleItems` etc.).
- **Onsite quick updates can text the assignee**: `getOnsitePrompts` resolves each prompt's textable recipients (direct company assignment, or the company filling an assigned role via `project_role_members`); the card's "Text …" panel prefills a per-trigger nudge and sends through the same `sendQuoTextToSub` action as the schedule dialog (server re-verifies the assignment).
- **Bulk copy to job**: the list view's bulk bar copies selected work items + to-dos to another project (`bulkCopyScheduleItems`): parent/child + predecessor edges survive when both ends are selected, statuses reset, checklists reset, assignments verbatim, milestones skipped, baseline stamped if the target is baselined.
- **Project History tab** (0073): `project_history` (staff-read, trigger-only writes via SECURITY DEFINER `record_project_history`) audits create/update/delete on schedule_items, decisions, daily_logs, project_files, project_payments, bid_packages, purchase_orders, project_members, project_role_members, schedule_assignments (+ projects updates) with per-field diffs and a `txid` batch key. `project_id` is a bare uuid (no FK) so project-delete cascades can't deadlock on it. Rendered at `/projects/[id]/history` (tab right of Roles, hidden from client/trade; latest 500, batched by txid).
- **Recently deleted / restore** (0088): `deleted_items` (staff-read, trigger-only writes, bare-uuid project_id like history) snapshots every DIRECT delete of schedule_items / decisions / daily_logs / project_files / bid_packages / purchase_orders — full row + children (subtree to-dos, choices, line items, attachments, assignments, checklists, comments, predecessor edges) + link metadata (due-date anchors, follow-up materializations) + storage paths. Cascade deletes are skipped via `pg_trigger_depth() > 1` (the parent's snapshot covers them; project deletes leave no trash). The capture trigger is named `trg_a_trash_*` so it fires before `trg_si_decision_due_unlink` (BEFORE triggers run alphabetically). Delete actions no longer remove attachment Storage objects — purge (0089/0090, `lib/trash-purge.ts`, run lazily on History-page load AND by the daily `/api/cron/trash-purge` sweep) is claim-batch (`purge_claimed_at` — a claimed entry can't be restored, killing the check-vs-remove race; stale claims re-sweep after 1h) → verify paths against live attachment tables (`unreferenced_storage_paths`) → remove Storage objects → finalize rows; a failed removal unclaims and retries next sweep, and 200-row batches loop until drained so PostgREST's 1,000-row cap can't truncate a backlog. Restore = `app/actions/trash.ts:restoreDeletedItem`: claim-then-insert via `claim_deleted_item` (idempotent; unclaimed on main-row failure), re-inserts under the caller's RLS session with fixups — missing FK targets nulled/skipped with warnings, NOT-NULL `created_by` falls back to the restorer, reused decision/bid/PO numbers reallocated via the advisory-lock RPCs, predecessor edges cycle-checked (`wouldCreateCycle`), file version-chain head repaired, `claim_restored_entities` closes overlapping bulk-delete entries. UI: "Recently deleted" toggle on the History tab (`trash-panel.tsx`). Payments are excluded — they keep their own soft-delete/restore (0022).

## Communications / Quo phone — model

- Every outbound text/call and inbound event is logged to `communications` (see the Daily Logs/Bids modules for feed rendering). `sendQuoSms` in `lib/quo.ts` is the single SMS sender; `app/api/inbound/quo/route.ts` ingests the OpenPhone webhook (`message.received/.delivered`, `call.completed`, `call.recording.completed`).

So automated texts (bid/PO/assignment/manual/hub-reply/AI-apply) go out from the initiator's own number with **zero call-site changes** — they all pass `log.sent_by`. An explicit `opts.from` still overrides.
- The Team picker is populated by `listQuoPhoneNumbers()` (`GET /v1/phone-numbers`); empty when `QUO_API_KEY` is unset, and the dialog then only offers the shared number. `updateProfile` maps a 23505 to a friendly "already assigned" error; a non-staff role clears the assignment.
- **Native attribution**: the webhook matches each event's business-side number (`phoneNumberId` first, then E.164) to a profile via `staffProfileForQuoNumber`. Outbound rows get `sent_by` = that staffer (so texts they type and calls they dial in the Quo app show their name in the feed, which renders `sent_by` for outbound); inbound rows keep the counterparty as author but stash the line owner in `meta.quoStaffProfileId`. App-sent texts already hold `(source,provider_id)` with richer attribution, so the webhook's `ON CONFLICT DO NOTHING` leaves them untouched. Note: the Quo/OpenPhone API can only *send texts* and *track* calls (via webhook) — it cannot *place* calls.
- **Compose** ("New message"): staff can START a text or email from either communications tab — `ComposeMessageButton` (`components/comms/compose-dialog.tsx`) → `composeMessage` in `app/actions/communications.ts`. Recipients resolve server-side (a company from the directory, or the job's client contact slots on the project tab; a `custom` raw number/address is the deliberate escape hatch). SMS rides `sendQuoSms` (staffer's own Quo number), email rides `sendEmail` (Outlook→Resend); both log as `manual_sms`/`manual_email` with company/project attribution — plus the client's portal profile (best-effort `project_members` match) so a client-directed message shows in their portal feed too. Project-tab sends file to that job; global-hub sends stay global unless the recipient is a job's client contact. Company contact lists come from `lib/comms/contacts.ts:buildCompanyContacts`. **Clients** get their own `ClientComposeButton` on the project tab (no recipient picker — always "the team"): `clientComposeMessage` verifies membership via the RLS-scoped project read, inserts the inbound `communications` row via the admin client (`source_kind='client_message'`, stamped with their profile+project so both sides' feeds show it), then fans out staff in-app notifications + email (the requestDueDateReset pattern).
- **Job attribution & the global hub**: `matchCounterparty` (`lib/comms/match.ts`, via the `match_contacts_by_phone/email` service-role RPCs in 0067) resolves an inbound number/address to a project when it can (project client contact → profile → **recent conversation** → company's single active engagement). The recency heuristic (`recentProjectForCompany`, phone traffic only) files a sub's reply to the job we last texted them about: all project-stamped rows for that company+number in the last 7 days must agree on ONE project, and the evidence must include an outbound/staff row (an auto-filed inbound can't feed itself). The hub offers "Re-file to a different job" on filed rows as the correction path; AI-sent texts stamp project_id when the company's schedule assignments are single-project. Quo phone traffic is **never queued for manual placement**: the webhook always writes `status='logged'`, so it auto-files to a job only when the matcher is confident (a single-job sub) and otherwise stays global-only. A conversation can span multiple jobs, so this is intentional — staff don't have to file anything. The global staff hub `/communications` (`GlobalCommunicationsClient`) is the searchable home for all channels (shows everything except `status='ignored'`); unfiled rows offer a quiet optional "File to job" (`assignCommunication`) / "Dismiss" (`ignoreCommunication`) — there is no nagging needs-review queue. Job-level `/projects/[id]/communications` still shows only `project_id=this AND status='logged'`. (Inbound email uses the same matcher but prefers the `+p_<project_id>` reply plus-tag; unmatched email keeps `status='needs_review'`, which now renders identically to any other unfiled row in the hub.)
