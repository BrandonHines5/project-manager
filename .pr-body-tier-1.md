## Summary

First of three PRs implementing the end-to-end review's recommendations. This one is **pure low-risk hardening**: nothing user-facing changes meaningfully, but durability, accountability, and a11y improve materially.

## Database hardening (migrations 0021–0025, already applied to HH-ProjectManager)

- **0021** — `schedule_items` / `daily_logs` / `decisions`: backfill 414 legacy NULL `created_by` rows to the project owner, install a before-insert trigger that fills NULL from `auth.uid()`, swap FK from `ON DELETE SET NULL` to `RESTRICT`, then enforce NOT NULL. Trigger keeps the constraint deploy-order-independent.
- **0022** — `project_payments`: soft-delete (`deleted_at` + `deleted_by`), new `payment_audit(action, before, after, actor_id)` table. Client RLS now hides soft-deleted rows.
- **0023** — `decision_followup_materializations` junction table replaces the fragile description-LIKE marker for re-approval idempotency. RPC `client_decide_decision` rewritten to use the junction.
- **0024** — Compound indexes on `schedule_assignments(schedule_item_id, profile_id)` and `(…, company_id)` for the trade-read RLS lookup.
- **0025** — `append_checklist_item` RPC: atomic position allocation under a row lock for the AI agent's checklist appends.

## Server actions

- `app/actions/payments.ts` — hard delete → soft delete, idempotent; new `restorePayment`; every create/update/delete/restore writes a `payment_audit` row.
- `app/actions/schedule.ts` — `saveScheduleItem` now sets `created_by: profile.id` explicitly (the 0021 trigger would catch it, but explicit is honest).
- `app/actions/decisions.ts` — `materializeFollowups` reads/writes the junction instead of pattern-matching the description.
- `app/actions/ai-agent.ts` — `applyPlanAction` now revalidates per-project (only fans out to the layout when an id-scoped mutation lacks a `project_id`); threads `actorId` into `applyPlan` so create-mutations populate `created_by`.
- `lib/ai/apply.ts` — `add_checklist_item` uses the atomic RPC; `applyPlan` takes `actorId`.
- `lib/auth.ts` — self-heal warns to the server log so a silent client-role downgrade surfaces before the user hits a permission wall.

## UI primitives

- `globals.css` — promoted hardcoded sidebar color to `--color-sidebar`; both desktop sidebar and mobile drawer now consume it.
- `components/ui/dialog.tsx` — **focus trap** + **focus restore**: Tab cycles inside the dialog; close puts focus back wherever it was before open. `role="dialog"` + `aria-modal` moved onto the panel.
- `components/ui/input.tsx` — `Field` gains an `error` slot (replaces hint, exposes `role="alert"` for screen readers); `Input`/`Textarea`/`Select` take an `invalid` flag that flips border + ring to danger.
- `components/ui/skeleton.tsx` (new) — pulsing placeholder for list/dashboard pages.
- `components/ui/badge.tsx` — bumped warning/danger/info/muted text tones a step darker for WCAG AA.
- `app/(app)/layout.tsx` — added a `sr-only` skip-to-main link so keyboard users can bypass the long sidebar/topbar nav.

## Deferred to a later PR

The review recommended splitting `components/decisions/decision-drawer.tsx` (2,008 lines) and `components/schedule/schedule-item-dialog.tsx` (1,347 lines) into ~5 sub-components each. Both were skipped from this PR — the state in those drawers is deeply interwoven (choices ↔ cost_items ↔ attachments ↔ followups ↔ comments, with `client_key`/`new:N` keys to bridge unsaved/saved boundaries) and a clean split is a multi-hour refactor with regression risk and zero user-visible payoff. Better candidate for a dedicated PR with focused review.
